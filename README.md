## 🔄 Потоки данных (Data Flows)

### 1. Процесс обработки входящих сообщений
Архитектура взаимодействия между пользователем, инфраструктурой Telegram, ботом, аналитическим API и базой данных при обработке каждого сообщения.

<p align="center">
  <img width="839" height="382" alt="Процесс обработки входящих сообщений" src="https://github.com/user-attachments/assets/5d82a808-cce9-4737-9e67-989a71b2ef32" />
</p>

**Жизненный цикл обработки запроса:**

1. **Отправка:** Пользователь (`User Telegram`) отправляет текстовое сообщение в чат. Оно поступает на официальный `Telegram server`.
2. **Получение (Long Polling):** Компонент `Telegram Bot` непрерывно опрашивает сервера мессенджера, отправляя `HTTPS GET`-запросы на эндпоинт `/getUpdates`. В ответ сервер возвращает массив новых событий в формате `JSON Response (Update[])`.
3. **Передача в API:** Бот извлекает текст сообщения и перенаправляет его в бэкенд-службу с помощью `HTTP POST`-запроса на внутренний эндпоинт `/api/Telegram/Message`.
4. **Классификация (ML Engine):** Внутри `API` контроллер вызывает метод классификации. Компонент `Core` прогоняет текст через пайплайн векторизации и классификатора, возвращая метку класса: `Label (negative, positive)`.
5. **Ответ боту:** `API` завершает обработку запроса и возвращает боту успешный статус `HTTP 200 Ok` вместе с объектом ответа `ClassifyMessage`, содержащим вердикт модели.
6. **Персистенция:** Одновременно с этим `API` выполняет сохранение сообщения, метаданных и результатов анализа в базу данных `PostgreSQL`. При успешной записи уровень БД возвращает статус `Success`.


### 2. Процесс регистрации
Архитектура и логика инициализации нового администратора системы через интерфейс мессенджера (Telegram-клиент) с последующим сохранением учетных данных в хранилище.

<p align="center">
  <img width="974" height="432" alt="Процесс регистрации" src="https://github.com/user-attachments/assets/19749d54-fc28-43df-96f7-c8c4f057d693" />
</p>

**Жизненный цикл обработки запроса на регистрацию:**

1. **Инициация:** Пользователь в приложении (`Telegram client`) вводит команду `/start` и передает свои учетные данные (`Login, password`). Информация поступает на `Telegram server`.
2. **Получение события (Long Polling):** Служба `Telegram Bot` запрашивает новые события с серверов мессенджера методом `HTTPS GET` через эндпоинт `/getUpdates`. Сервер возвращает событие в формате `JSON Update`.
3. **Запрос к бэкенду:** Бот парсит полученный JSON, формирует объект запроса `SetAdminRequest` и отправляет его на шлюз бэкенда с помощью `HTTP POST` на специализированный эндпоинт `/api/Auth/SetAdmin`.
4. **Запись в БД (Персистенция):** Компонент `API` принимает запрос, обрабатывает данные и инициирует транзакцию добавления нового администратора в базу данных `PostgreSQL`. База данных возвращает ответ `Success`.
5. **Подтверждение бэкенда:** После успешной фиксации данных в БД, `API` отвечает боту успешным статусом `HTTP 200 Ok`, прикрепляя к ответу объект `AdminRegistrationResponse`.
6. **Уведомление пользователя:** `Telegram Bot` формирует финальный ответ для администратора и отправляет его обратно на сервера мессенджера через `HTTPS POST` на эндпоинт `/sendMessage`. Сервер подтверждает доставку (`JSON Response (Status 200)`), и пользователь видит в интерфейсе сообщение об успешной регистрации.


### 3. Процесс аутентификации в WPF-приложении
Архитектура верификации прав доступа при входе администратора в десктопную панель управления (Dashboard).

<p align="center">
  <img width="974" height="102" alt="Процесс аутентификации в WPF-приложении" src="https://github.com/user-attachments/assets/d1ce8d31-ef67-494e-a187-0319e5882b20" />
</p>

**Жизненный цикл обработки запроса на аутентификацию:**

1. **Ввод данных:** Администратор (`User WPF`) вводит свои учетные данные (`Login, Password`) в форму авторизации десктопного приложения.
2. **Передача на бэкенд:** Клиент `WPF` упаковывает данные в объект запроса `LoginRequest` и отправляет его по протоколу HTTP методом `POST` на эндпоинт удаленного сервера `/api/Auth/Login`.
3. **Проверка в хранилище:** Компонент `API` принимает запрос и выполняет сверку переданных данных с хранилищем, отправляя запрос в базу данных `PostgreSQL`.
4. **Ответ хранилища:** База данных возвращает профиль администратора, если совпадение найдено, либо пустое значение (`Admin Data (or NULL)`), если данные неверны.
5. **Авторизация:** На основе ответа от БД уровень `API` формирует финальный ответ для клиента: при успешной проверке возвращается статус `HTTP 200 Ok` вместе с объектом `AdminRegistrationResponse`, предоставляя доступ к панели.

Пайплайн динамического запроса и агрегации системных метрик для визуализации графиков и отчетов в реальном времени.

**Жизненный цикл формирования аналитики:**

1. **Инициация:** Пользователь панели управления (`User WPF`) переходит во вкладку статистики или обновляет текущий экран.
2. **Запрос данных:** Приложение `WPF` генерирует асинхронный `GET`-запрос на аналитический эндпоинт бэкенда `/api/Stats`.
3. **Агрегация из БД:** Компонент `API` обрабатывает входящий запрос и обращается к слою данных, отправляя запрос для расчета метрик, выборки топа токсичных пользователей или логов за период в `PostgreSQL`.

Реляционная модель данных, спроектированная по подходу *Code First (Entity Framework Core)*. База данных содержит 4 основные сущности для обеспечения сквозного логирования, авторизации и управления конфигурациями чатов.

### Описание сущностей и связей:

#### 1. Messages (Сообщения)
Хранит историю всех перехваченных текстовых сообщений и результаты их скоринга ML-движком.
* `Id (PK)` — уникальный идентификатор записи.
* `TextLabel` — текстовая метка классификации (результат предсказания: *positive* / *negative*).
* `Confidence` — уровень уверенности (вероятность) модели в вынесенном вердикте.
* `Created_at` — штамп даты и времени получения сообщения.
* `Chat_id (FK)` — связь с таблицей чатов (многие к одному).
* `User_id (FK)` — связь с таблицей пользователей (многие к одному).

#### 2. Chats (Модерируемые чаты)
Содержит информацию о Telegram-каналах и группах, в которых работает бот.
* `Id (PK)` — внутренний идентификатор чата в системе.
* `Chat_id_tg` — реальный уникальный ID чата в инфраструктуре Telegram.
* `Name` — название группы или канала.
* `Added_at` — дата подключения чата к системе мониторинга.
* `Admin_id (FK)` — связь с таблицей администраторов, привязавших данный чат (многие к одному).

#### 3. Users (Пользователи Telegram)
Участники чатов, чья активность анализируется системой на предмет токсичности.
* `Id (PK)` — внутренний идентификатор пользователя.
* `User_id_tg` — реальный ID аккаунта в Telegram.
* `Username` — публичный никнейм пользователя (аккаунт вида `@username`).
* `First_name` — имя пользователя, указанное в профиле.

#### 4. Admins (Администраторы системы)
Учетные данные для авторизации в десктопной панели управления (`WPF Dashboard`).
* `Id (PK)` — уникальный идентификатор администратора.
* `Login` — уникальное имя пользователя для входа.
* `Password` — пароль для аутентификации.
* `TelegramId` — привязанный аккаунт Telegram для управления через бота.

#### TgMessageRequest
*Описание:* Пакет данных входящего сообщения из Telegram для последующего анализа.
```json
{

  "firstName": "Имя"

```
* **text** (`string`): Полный текст отправленного сообщения.
* **chatIdTg** (`long`): Уникальный идентификатор Telegram-чата (группы/супергруппы).
* **chatName** (`string`): Публичное название или заголовок чата.
* **userIdTg** (`long`): Уникальный идентификатор автора сообщения в Telegram.
* **username** (`string`): Никнейм пользователя в Telegram (без символа @).
* **firstName** (`string`): Имя пользователя, указанное в его профиле.

## 🚀 Быстрый старт (Quick Start)

### 📋 Предварительные требования
* **.NET 8 SDK** или выше
* СУБД **PostgreSQL**
* Токен Telegram-бота от `@BotFather`

---

### 🛠 Инструкция по развертыванию

#### 1. Клонирование репозитория
```bash
git clone https://github.com/Pavel-Sedlovec/Negative_classifier.git
cd Negative_classifier
```

#### 2. Настройка конфигурации (User Secrets)
Все конфиденциальные данные вынесены в локальное хранилище **User Secrets** для предотвращения утечек в репозиторий.

**А) Строка подключения к PostgreSQL (проект API):**
```bash
cd TelegramToxicMonitor.API
dotnet user-secrets init
dotnet user-secrets set "ConnectionStrings:DefaultConnection" "Host=localhost;Database=negative_db;Username=postgres;Password=your_password"
cd ..
```

**Б) Токен Telegram-бота (проект Bot):**
```bash
cd TelegramToxicMonitor.Bot
dotnet user-secrets init
dotnet user-secrets set "BotConfiguration:BotToken" "1234567890:ABCdefGhIJKlmNoPQRsTUVwxyZ"
cd ..
```

#### 3. Применение миграций базы данных
Выполните команду в корневом каталоге решения для автоматического создания структуры таблиц:
```bash
dotnet ef database update --project TelegramToxicMonitor.Data --startup-project TelegramToxicMonitor.API
```
*(Если утилита `dotnet ef` не установлена, предварительно выполните: `dotnet tool install --global dotnet-ef`)*

#### 4. Запуск компонентов системы

Запуск **Backend API** (серверная часть и ML-движок):
```bash
dotnet run --project TelegramToxicMonitor.API
```

Запуск **Telegram-бота** (обработка сообщений через Long Polling):
```bash
dotnet run --project TelegramToxicMonitor.Bot
```

Запуск **Панели администратора** (WPF Dashboard):
```bash
dotnet run --project TelegramToxicMonitor.WPF
```

---

### 🏋️‍♂️ Обучение модели (Опционально)
Для переобучения классификатора SVM или запуска алгоритма сжатия выборки STOLP выполните:
```bash
dotnet run --project TelegramToxicMonitor.Trainer
```
Файлы весов автоматически обновятся в директории Core-движка.
